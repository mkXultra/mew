M6.24 Review Artifact

Native Tool Loop の現在構造と理想構造

目的は、implement_v2 を Codex 風の provider-native tool/function calling loop として保ちながら、 mew 固有の replay、context 圧縮耐性、step analysis、acceptance safety をどこに置くべきかをレビュー可能にすること。

結論は、harness は protocol/safety runtime、WorkFrame は attention/projection layer、 CompletionResolver は完了判定として分けるのが一番筋が良い。

Executive Summary

現在の問題

native loop は provider-native transcript を使い始めているが、finish_call の扱いと semantic completion の責務境界がまだ曖昧。さらに現在の harness は _native_final_verifier_closeout、structured_finish_gate、 native_loop_control などの semantic-ish control も一部持っている。

責務境界が曖昧 finish が早い

理想の方向

harness は tool call / output の正しさと安全境界だけを守る。WorkFrame は compact card として次 turn に注入し、 model の判断材料を整える。現在 harness / exec runtime に散っている完了判定は CompletionResolver に寄せる。

native loop 維持 WorkFrame 注入

現在の Native Tool Loop

現在の本線は src/mew/commands.py から run_live_native_implement_v2 を呼び、native_tool_harness.py が provider-native response items を transcript に保存し、tool を実行する構造。

Current implement_v2 native loop commands.py | | selects implement_v2 native transcript runtime v native_tool_harness.py | | _request_descriptor() | - task_contract | - persisted_lane_state | - prior native transcript items | - small native_loop_control only when thresholds fire v OpenAI Responses / Codex Web API | | provider-native response items v NativeTranscript | | function_call / custom_tool_call / finish_call v tool router | | read_runtime / write_runtime / exec_runtime v ToolResultEnvelope | | function_call_output / custom_tool_call_output / finish_output v NativeTranscript artifacts | | response_transcript.json | response_items.jsonl | proof-manifest.json v fastcheck / replay / step analysis
現状の WorkFrame は native harness の主経路ではなく、主に projection、fastcheck、replay、旧 v2 runtime 周辺の概念として残っている。 一方で harness には finish_call acceptance、_native_final_verifier_closeout、 native_loop_control のような semantic control が入り込んでいる。つまり現在は 「WorkFrame は主経路に薄いが、harness は semantic に厚い」という歪んだ中間状態。

_native_loop_control_state first-write / verifier-repair などの制御カード next_action_policy を含むため、planner 的 policy が harness に混ざる drift risk がある
_native_final_verifier_closeout finish 後の verifier closeout。status を completed / blocked に上書きし得る CompletionResolver 的な意味判定が harness 内にあるため、移行時に責務を明示して移す必要がある

理想の Native Tool Loop

理想は、raw/native transcript を source of truth にしつつ、WorkFrame を「次 turn の注意カード」として注入する形。 WorkFrame は tool 実行や完了判定を握らず、model が次 action を決めるための compact projection だけを提供する。

Ideal implement_v2 native loop NativeTranscript source of truth | +--> ProviderAdapter / RequestBuilder | - build provider-safe input items | - lower tool schemas | - preserve native ResponseItem pairing rules | +--> Harness | - validate provider-native item pairing | - execute tools | - enforce permissions and allowed roots | - persist tool outputs and side effects | +--> HarnessWorkFrameProjection | - compact goal / latest actionable / evidence refs | - expected artifact and verifier obligations | - finish readiness risk | - step-shape metrics | +--> CompletionResolver | - accepts or blocks semantic completion | - uses finish claim, typed evidence, WorkFrame, replay hints | - produces lane_status independent from transcript pairing | v Next provider request | | input = | 1. task contract | 2. native transcript window | 3. compact_sidecar_digest with bounded WorkFrame projection fields | 4. tool schemas v Model decides next tool call naturally
この構造なら、Codex 風の native tool loop を保ったまま、mew 固有の常駐性、replay、context 圧縮耐性、dogfood proof を追加できる。 provider-visible object は新設せず、既存 request surface の compact_sidecar_digest に bounded WorkFrame fields を入れる。

Guard の置き場所

Harness に残す guard

  • call_id / output pairing
  • arguments JSON validation
  • unknown tool rejection
  • allowed read/write roots
  • write approval
  • shell permission
  • max_turns / timeout / transport failure

Harness から出す guard

  • この task が本当に完了したか
  • 外部 verifier の oracle を満たしたか
  • source grounding が十分か
  • 次に patch すべきか、probe すべきか
  • repeated failure の修復方針
  • task-specific artifact 判定
finish_call は transcript 上では正しく pair されてよい。ただし lane status を completed にするかは CompletionResolver が決める。これにより harness は protocol runtime として薄く保てる。

Finish State Machine

レビュー指摘で最も重要だったのは、finish_call の transcript protocol と lane completion が混ざっている点。 理想構造では、finish は以下の状態遷移として扱う。

finish_call state machine model emits finish_call | v harness validates arguments | v CompletionResolver evaluates semantic completion | +-- completed | lane_status = completed | paired finish_output contains allow result | +-- blocked_continue | lane_status remains active | paired finish_output contains block result + blockers | mirror resolver_decision to sidecar/artifact | next request receives raw transcript + WorkFrame blockers | +-- blocked_return lane_status = blocked return to supervisor/reentry with concrete blockers

これにより、harness は provider-native pairing を守り、CompletionResolver は semantic state を決める。 verifier 実行そのものは harness / tool runtime closeout に残すか、blocked_continue の次 action として model に返す。 CompletionResolver は実行済みの _native_final_verifier_closeout / structured_finish_gate evidence を消費して判定する。 resolver_decision は native ResponseItem ではなく sidecar artifact。provider-visible には paired finish_output の content と次 turn の compact_sidecar_digest で伝える。 現行の単一 status="blocked" は、closeout 後の停止なら blocked_return、 finish rejection から同一 lane 継続できるなら blocked_continue に分割する。

WorkFrame の責務

WorkFrame は planner ではなく、model の attention を整える projection。命令ではなく、判断材料を compact に提供する。 ただし現行 workframe.py は required_next と forbidden_next をすでに持つ。 そのため、この文書の「理想 WorkFrame」は現行 schema のそのまま維持ではなく、native loop 向けに縮約した target。

現行フィールド 理想での扱い レビュー観点
goal 維持。タスク目的、成功条件、制約を短く渡す task prompt とズレていないか
latest_actionable latest_actionable として維持。ただし action 強制にしない model が raw transcript を辿らず最新問題を把握できるか
required_next 現行は prescriptive。理想では普通の repair 指示から外し、必要な情報は attention_hints / tool_context に縮約。finish-ready、verifier-closeout、blocked、明示的 transition_contract 互換だけ nullable limited field として許容。 Codex 風の model autonomy を殺していないか
forbidden_next safety / repetition guard として限定維持 policy 過多ではなく、失敗ループ防止に限定されているか
changed_sources 編集済み source と verifier freshness の補助情報として維持 raw transcript の代替ではなく index になっているか
verifier_state fresh / stale / failed verifier の compact state として維持 古い failure を引きずっていないか
finish_readiness allow/block ではなく risk、blockers、missing obligations を示す 外部 oracle と内部 verifier の差を見える化できるか

避けたい形: WorkFrame が通常 repair で required_next: edit vm.js line 123 のように action を強制すること。 これは model の判断能力を殺し、transition_contract が厚くなりすぎた問題に戻る。
良い形: finish_readiness.blockers = ["external_artifact_missing"]、 attention_hints = ["latest passing verifier checked frame.ppm, but task/external verifier expects /tmp/frame.bmp"] のように 判断材料を提示すること。

今回の make-mips-interpreter 失敗に当てると

直近の native 10min diagnostic では、mew は frame.ppm を生成して自己検証し、 外部 verifier が期待する /tmp/frame.bmp を作らずに finish した。

Observed failure Model: node vm.js -> frame.ppm exists -> finish External verifier: expects /tmp/frame.bmp FileNotFoundError: /tmp/frame.bmp Current gap: harness paired finish correctly, current closeout / finish gates did not know or enforce this external oracle obligation early enough. Ideal handling: harness records finish_call and finish_output WorkFrame projects: finish_readiness = risky missing_obligations = ["external_artifact:/tmp/frame.bmp"] latest verifier checked frame.ppm instead of verifier-visible artifact CompletionResolver blocks lane completion next turn receives compact card and can repair artifact target

これは harness に /tmp/frame.bmp 固有 guard を入れる話ではない。WorkFrame と CompletionResolver が、 task contract、execution contract / oracle bundle、typed evidence、fresh verifier evidence を入力にして、 「finish evidence が external oracle と一致していない」ことを model-visible かつ replay-visible にする話。

Recommended Decision

現時点では、WorkFrame を harness に埋め込まず、compact_sidecar_digest 内の bounded WorkFrame projection として注入するのがよい。 harness は protocol/safety だけを守り、semantic completion は CompletionResolver へ分離する。

Recommended boundary Harness: provider-native transcript runtime tool routing permission/safety proof artifacts Tool Runtime: execute read/write/exec tools emit typed evidence and side effects run deterministic closeout when explicitly requested WorkFrame: compact projection attention hints evidence refs finish risk CompletionResolver: semantic completion decision typed evidence and external oracle alignment lane_status = completed | blocked_continue | blocked_return Model: reads raw transcript + WorkFrame decides next native tool call
この decision の実装上の意味は、「harness に semantic guard をさらに足す」ではない。 既に harness / exec runtime にある semantic-ish controls を CompletionResolver へ移し、 WorkFrame は resolver と model のための bounded projection にすること。 blocked_continue は lane を active に保ち、paired finish_output と sidecar の blockers を次 turn に渡す。 native_loop_control.first_write_due / verifier_repair_due は bounded signals として WorkFrame の attention_hints に折り込み、prescriptive next_action_policy と命令文は落とす。

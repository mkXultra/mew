M6.24 Internal Finish Gate レビュー用日本語HTML

対象は implement_v2 の完了判定を、main model に見える finish tool から、通常の最終応答後に走る mew 内部の finish gate へ移す設計です。目的は Codex-like hot path に寄せつつ、 完了判定・検証・証跡を sidecar 側で厳密に保つことです。

Status: design only 出力: review-only HTML 元Markdownは未変更 実装コード変更なし

1. 1分で分かる要約

何を変えるか

main model に finish tool を呼ばせない。モデルは通常の assistant final response を出し、controller が done_candidate を記録してから内部 finish gate を走らせる。

何を隠すか

task_contract、finish gate/resolver JSON、 evidence_refs、planner rationale などの mew 内部完了スキーマを main model の hot path に出さない。

どう閉じるか

OK ならモデルの最終応答を保って lane completed。NG continue なら抽象的な修復信号で再開。 NG return なら安全・権限・予算・plateau などで blocked として返す。

レビューの中心論点: provider-visible completion protocol をなくしても、完了判定の厳密さ、verifier precedence、 proof/replay/observability が落ちない設計になっているかを確認する。

3. なぜこの変更をするのか

make-doom-for-mips のズレとの関係

このHTMLでは、元設計の趣旨を次のように読む。make-doom-for-mips のような 実タスクでズレが出る場合、main model が「mew 固有の完了スキーマを満たすこと」に引っ張られると、 本来の coding hot path から意識が逸れる。設計は、モデルには普通に作業させ、完了判定の厳密さは controller/sidecar 側で担保する方向へ寄せている。

重要なのは、モデルに隠し期待値や gate の JSON を学習させて解かせるのではなく、 モデル可視の事実と通常の verifier 実行から修復させること。NG continue でも raw gate blocker ではなく、 一、二文の修復焦点と安全な refs だけを返す。

Codex-like hot path に寄せる意図

Codex-like hot path では、モデルに見える状態を限定し、編集・テスト・通常応答を中心にする。 WorkFrame、proof、typed evidence、resolver/gate details は sidecar-first に置く。 この設計は finish tool を provider-visible surface から外し、 hot path から completion-specific schema pressure を取り除く。

5. Verifier precedence

  1. まず active managed commands を close または observe する。
  2. configured_verifier が明示供給され安全なら第一 authority。これで証明できたら planner を completion authority として起動しない。
  3. finish_verifier_planner は第二 authority。configured verifier が証明しない場合に、planner-visible inputs だけで request を作る。
  4. auto_detected_verifier は原則 diagnostic-only。planner rejection を黙って置換して completion authority になってはいけない。
  5. auto-detected completion authority は、allow_auto_detected_finish_verifier_fallback_v1 のような明示 policy がある場合だけ許可される。
入力 planner に渡してよい planner に渡してはいけない
タスク情報 main model に元々渡した raw user task text raw task_contract JSON、hidden acceptance constraints
verifier ユーザー/task/lane config で明示供給された configured verifier command hidden benchmark metadata から推測した expected output や失敗詳細
実行結果 model-visible tool results、artifact refs、可視 transcript summaries private resolver/gate blocker JSON、proof oracle obligation records
runtime facts workspace root、allowed roots、remaining timeout、command lifecycle、visible mutation summary モデルに見えていない oracle 制約や benchmark-only expected artifacts
重要制約: gate は hidden external verifier/oracle state を内部判断に使ってよいが、 finish_verifier_planner へ hidden oracle knowledge を shortcut として渡してはいけない。

7. Close Gate チェックリスト

  • production native provider requests に finish tool descriptor がない。
  • finish argument schema fields が provider request にない。
  • main-model input に raw task_contract、gate/resolver JSON、planner rationale、proof manifest body がない。
  • planner request は planner-visible inputs だけで構成される。
  • planner request に hidden verifier detail、hidden acceptance、raw contract、oracle constraint、benchmark-only expected output がない。
  • sidecar で configured verifier → planner → diagnostic-only auto の precedence が見える。
  • auto-detected verifier が rejected planner command を黙って置換しない。
  • no-tool assistant final response から done candidate がちょうど1つ作られる。
  • すべての done candidate に internal finish-gate decision がちょうど1つある。
  • production gate/planner rows は done_candidate_id で keying される。
  • OK は lane completed にし、モデルの final response を保持する。
  • NG continue は抽象 signal で再開し、内部 schema を漏らさない。
  • NG continue の hard cap と repeat plateau rule が守られる。
  • NG return は supervisor-safe summary で blocked として止まる。
  • planner decisions は sidecar-only で manifest-hashed。
  • compact digest は done-candidate/internal-gate fields を使い、legacy finish-readiness fields を含まない。
  • response_transcript.json、sidecars、proof-manifest.json で completion outcome が決定的に replay できる。
  • hot-path step-diff が model work、done candidate、internal gate、NG resume を区別できる。

8. 実装時に drift しやすいポイント

  • finish を tool_policy.py から消しても、registry/routes/renderer/schema/provider adapter に残って provider-visible surface へ復活する。
  • done_candidate_id へ rekey したつもりで、planner/gate の一部 artifact がまだ finish_call_id production key になる。
  • NG resume signal に raw gate blocker、task_contract field name、evidence_refs などが混ざる。
  • planner request builder が hidden external verifier detail や benchmark-only expected output を便利情報として渡してしまう。
  • auto_detected_verifier が diagnostic-only を超えて、planner rejection 後の silent fallback になる。
  • ambiguous no-tool prose で continuation を無制限に出し、done candidate に収束しない。
  • observability/fastcheck/step-diff の追加が artifact-only ではなく live path を変える。
  • legacy run_live_json_implement_v2() の finish tests が production native close gate を満たした扱いになる。

9. ユーザーがレビューするときの質問リスト

  1. main model に見える入力は、本当に通常の coding loop と安全な NG resume signal だけになっているか。
  2. finish removal は tool policy だけでなく registry、routes、renderer、schema、provider adapter まで閉じているか。
  3. done_candidate の検出条件は、final response と途中の no-tool prose を保守的に分けられるか。
  4. 曖昧 no-tool prose の「1回だけ continuation、その次は no_tool_repeat candidate」は十分に実装可能か。
  5. internal gate は task_contract を読める一方、main model と planner へ漏らさない境界がテストできるか。
  6. configured verifier がある場合に planner を不要にする precedence は、sidecar 上で後から監査できるか。
  7. auto-detected verifier の fallback policy は、許可条件・記録項目・拒否時挙動が十分に明示されているか。
  8. NG continue の cap と plateau signature は、無限修復ループを止めるのに足りるか。
  9. OK 時に controller metadata を user response に混ぜず、モデルの final response を保持できるか。
  10. legacy replay/quarantine と production native route の境界は、テスト名・artifact key・audit で混同されないか。
  11. make-doom-for-mips のような実タスクで、mew 固有完了スキーマへの適応ではなく、普通の編集・検証へ戻れる設計になっているか。
  12. Close Gate は「実装済みっぽい」ではなく、provider request artifacts と sidecar hash で機械的に判定できるか。

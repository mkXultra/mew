# M5.1 Close Gate 2026-04-20

Status: passed.

This document closes Milestone 5.1, Trust & Safety Close-Out. M5.1 did not
move the M5 self-improvement gate. It hardened the already-closed M5 loop so
future self-improvement is harder to approve accidentally and easier to reject
correctly.

## Gate Source

`ROADMAP.md` defines M5.1 done when:

- a proposed self-improvement loop can be adversarially reviewed for
  product-goal drift, safety-boundary violations, weak evidence, and missing
  verification;
- governance edits, permission/policy edits, external-visible side effects,
  budget exhaustion, and ambiguous recovery are blocked or escalated by
  mechanical hooks rather than prompt text alone;
- at least one self-improvement loop or equivalent scenario exercises the
  verifier and safety hooks with a readable audit bundle;
- refactor work remains limited to the code needed for the verifier and hooks.

## Interpretation

M5.1 is done because the safety boundary is no longer only prompt text. The
system now has review criteria, pre-execution blocking for known external side
effects, auto-approval escalation for governance/policy edits, and audit
escalation for budget and recovery boundary states.

The remaining risks are bounded and belong to later milestones:

- the external-visible command marker list is conservative, not exhaustive;
- budget/recovery handling may later gain richer policy actions;
- future self-improvement loops should use the verifier routinely, not only at
  milestone close.

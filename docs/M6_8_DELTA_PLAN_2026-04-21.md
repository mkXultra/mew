# M6.8 Delta Plan

Date: 2026-04-21  
Status: prep only. No M6.8 implementation starts until M6.7 closes and this
delta plan is reviewer-approved.

## Purpose

Turn the registered M6.8 roadmap entry into an implementation-ready split
without changing the active milestone.

Ordering remains:

1. finish M6.7 supervised 8-hour proof
2. approve this delta plan
3. land M6.8 as bounded supervised iterations

This document is intentionally operational. It defines:

- what the selector is allowed to do
- what must stay reviewer-owned
- what proof is required before chaining counts

## Scope

M6.8 is not unattended autonomy. It is supervised task chaining.

The selector may:

- read current roadmap/status/task state
- propose one next bounded roadmap task
- explain `selector_reason`
- reference the previous completed task

The selector may not:

- auto-start the next iteration without reviewer approval
- edit roadmap/status/governance files
- close milestones
- rewrite its own task-selection policy during the run

## Deliverables

### D1. Selector Input Contract

Goal:

- define the exact state the selector reads before proposing a next task

Minimum inputs:

- active milestone and its unmet done-when items
- open bounded roadmap candidates
- last completed iteration outcome
- recent blocker history
- explicit non-goals for the session

Proof:

- selector output is reproducible from the same input snapshot

### D2. Selector Output Schema

Goal:

- standardize the chained-task proposal

Minimum fields:

- `previous_task_id`
- `proposed_task_id` or `proposal_shape`
- `selector_reason`
- `scope_fence_check`
- `proof_plan`
- `why_now`
- `why_not_other_candidates`

Proof:

- reviewer can approve, reject, or edit the proposal without free-form parsing

### D3. Reviewer-Gated Handoff

Goal:

- next task does not start until reviewer approval is explicit and recorded

Minimum behavior:

- approved proposal becomes the next bounded iteration
- rejected proposal records the rejection and leaves control with reviewer
- edited proposal preserves the original selector proposal for audit

Proof:

- one rejection occurs naturally and the following approved task continues the
  chain cleanly

### D4. Chain Identity and Audit Trail

Goal:

- every chained iteration is auditable as part of one supervised run

Minimum fields per iteration:

- `previous_task_id`
- `selector_reason`
- reviewer decision
- drift canary result
- proof artifact path

Proof:

- a chained run can be reconstructed without chat history

### D5. Selector Scope Fence

Goal:

- extend M6.7 scope fencing to the selector itself

Minimum rules:

- selector output cannot target roadmap/status/governance files
- selector cannot approve itself
- selector cannot bypass proof-or-revert

Proof:

- one attempted governance-violating selector output is rejected and logged

### D6. Hard Human Checkpoint

Goal:

- cap supervised chaining before drift accumulates

Initial rule:

- maximum 3 consecutive automatic selections per supervised run

Proof:

- a fourth automatic selection request is blocked and logged as requiring a
  human checkpoint

## Proposed Iteration Split

Implement as separate bounded iterations after M6.7 closes:

1. D1 selector input contract
2. D2 selector output schema
3. D3 reviewer-gated handoff
4. D4 chain identity and audit trail
5. D5 selector scope fence
6. D6 hard human checkpoint

Keep them separate. Do not bundle selector generation, governance, and audit
into one pass.

## Required Proof Artifacts

Before M6.8 code starts, define:

- one canonical chained-run JSON artifact format
- reviewer decision log format for selector approval/rejection/edit
- one reduced drift canary suitable for every chained iteration
- one governance-violation fixture for selector scope-fence rejection

Required proof after implementation:

- three consecutive bounded iterations in one supervised run
- reviewer approval recorded on each step
- at least one natural reviewer rejection
- green drift canary throughout
- zero rescue edits

## Open Questions To Resolve Before Code

1. Should selector candidates come only from predeclared roadmap tasks, or may
   the selector propose a new bounded task shape that the reviewer then mints?
2. Where should `selector_reason` live so it is visible in both CLI surfaces
   and durable audit artifacts?
3. Which existing proof artifact format from M6.7 should be extended versus
   replaced for chained runs?
4. What is the smallest reduced drift canary that is still honest enough to run
   before every chained iteration?

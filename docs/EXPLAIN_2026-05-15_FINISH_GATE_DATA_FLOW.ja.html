finish gate とタスク要求のデータフロー

この資料は、Terminal-Bench の task instruction が mew の内部でどの形に変換され、最終的に finish 判定へどう使われるかを説明するためのメモです。特に make-doom-for-mips で起きた 「モデルには要求が見えているのに、finish gate の判定が弱くなる」問題を扱います。

結論

問題は「Harbor runner がタスク要求をまったく渡していない」ことではありません。 モデル入力には要求文が届いていました。問題は、finish gate が判定に使う task_description へ変換するとき、経路によって goal / objective が落ちうることです。

つまり、LLM は「何を作るべきか」を読めていても、完了判定側が「何を満たしたら完了か」を 弱く解釈する可能性がありました。

全体の流れ

1. Harbor instruction Terminal-Bench の自然文タスク要求
2. mew work --oneshot --instruction として CLI に渡る
3. task record task.description に保存
4. task_contract description, goal, objective, verify_command
5. model input LLM が読む Task / Objective
6. finish gate 完了してよいかを判定

重要なのは、モデル入力 と finish gate の判定入力 が同じものではないことです。 モデルに要求が見えていても、finish gate 側の task_description が不足していれば、 gate は誤って「十分な証跡がある」と判断します。

make-doom で実際に見えていたもの

今回の provider input には、次のような内容が入っていました。

Task
Title: One-shot mew work task
Objective: I have provided /app/doomgeneric/, the source code to doom...
I've also wrote a special doomgeneric_img.c ...
vm.js ... expect a file called doomgeneric_mips ...
After running `node vm.js` I expect that stdout will be printed appropriately,
and frames will be written to the file system.
Guidance: selected_lane=implement_v2 write_integration_observation_detail=true
Verifier: node vm.js

これは「モデルには要求が渡っていた」証拠です。したがって主原因は mew_harbor_runner.py の起動方法そのものではありません。

finish gate の動き

LLM calls finish(...)
  |
  v
native harness builds:
  - action: outcome, summary, evidence_refs, acceptance_checks
  - session: tool results, typed evidence, transcript
  - task_description: task_contract を文字列化した判定用テキスト
  |
  v
acceptance_done_gate_decision(task_description, action, session)
  |
  +-- typed evidence gate
  |     evidence_refs / evidence_events / oracle_bundle を解決
  |
  +-- legacy string safety gate
        task_description と acceptance_checks と tool output text から
        追加の安全判定を行う
  |
  v
allow_complete or block_continue
層 入力 役割
typed evidence gate evidence_refs, evidence_events, oracle_bundle 構造化された証跡が、finish claim を支えているかを確認します。
legacy string safety gate task_description, acceptance_checks, tool output text まだ typed 化されていない要求を文字列から推定し、危ない完了を止めます。

今回の弱点

修正前

task_contract = {
  "description": "...",
  "goal": "run node vm.js and render frames",
  "objective": "stdout and frames should be correct"
}

task_description used by gate:
  title + description + guidance + verify_command

goal/objective may be missing

gate が goal / objective を見ない場合、 frame.bmp の存在だけで runtime artifact の一部を満たしたように扱いやすくなります。

修正後

task_description used by gate:
  title
  + goal
  + objective
  + description
  + guidance
  + verify_command
  + acceptance constraints

gate 側も「stdout が適切」「frame が正しく描画される」という要求を拾えるため、 単なるファイル存在では完了しにくくなります。

なぜ finish command だけでは足りないか

LLM が作る finish verifier command は「何を実行して観測するか」を決めます。 しかし「その観測がタスク要求を満たしているか」は別問題です。 

finish verifier command:
  node vm.js

observed:
  /tmp/frame.bmp exists
  stdout was printed

still missing:
  stdout is the expected one
  frame visually matches what the task verifier expects
  the produced artifact is not stale from a previous run

したがって、finish command の exit code が 0 でも、finish gate は 「artifact existence」「runtime freshness」「visual quality」を分けて確認する必要があります。

make-doom の false finish はなぜ起きたか

model:
  "node vm.js ran, /tmp/frame.bmp exists, finish"

native harness:
  final verifier closeout runs node vm.js
  closeout evidence proves artifact freshness/existence

finish gate:
  task_description is weak
  visual-quality obligation is not strong enough

resolver:
  closeout resolves "runtime_final_verifier_artifact_evidence"
  finish is allowed

external verifier:
  frame similarity is too low
  reward = 0

つまり、外部 verifier が見ている「画面品質」と、内部 finish gate が見ていた「成果物の存在」がずれていました。

今回入れている修正の意味

修正 意味
goal / objective を task_description に含める finish gate が本来の要求を見落とさないようにします。
visual task では artifact-only closeout を許さない /tmp/frame.bmp があるだけでなく、画面品質に関する証跡を要求します。
fastcheck の tool output TODO false positive を修正 source code 内の // TODO を WorkFrame/Todo state leak と誤判定しないようにします。

理想形

長期的には、finish gate が task_description 文字列を読む量を減らし、 task contract を構造化して渡す方がよいです。 

typed task contract:
  verifier_command: "node vm.js"
  expected_artifacts:
    - path: "/tmp/frame.bmp"
      produced_by: "verifier_command"
      freshness: "must be created by final verifier run"
      quality:
        kind: "visual_similarity"
        external_verifier: true
  output_obligations:
    - stdout must match expected runtime behavior

この形になれば、finish gate は文字列から推定するのではなく、明示された obligation を解決できます。 今の修正は、その理想形へ移るまでの安全策です。
